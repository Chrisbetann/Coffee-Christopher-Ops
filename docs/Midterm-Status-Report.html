1. Project Overview

The Coffee Christopher Ops Suite is a full-stack, QR-code-based digital ordering and operations management system designed for a coffee shop environment. Customers scan a QR code to browse the menu, customize items, and place orders from their phones. Staff manage the order queue, menu, inventory, and view sales analytics through a password-protected admin dashboard. The system is built as a monorepo with a React frontend and a Node.js/Express backend connected to a PostgreSQL database hosted on Supabase.

Reviews will be displayed publicly on item detail pages and moderated by admin. This feature directly improves customer trust and gives the shop owner actionable feedback on product quality.

6. Problems Encountered

Several common challenges were encountered throughout development, particularly as this project represents the developer's first experience building a full-stack application with a relational database.

Problem 1 – CORS Errors Blocking Frontend-to-Backend Communication

When the React frontend first attempted to fetch menu data from the Express server, the browser blocked all requests with: "Access to XMLHttpRequest has been blocked by CORS policy." This was caused by the Express server not having cross-origin resource sharing configured.

Fix: Installed the cors npm package and added app.use(cors({ origin: process.env.CLIENT_URL })) in server/index.js. Also added the correct CLIENT_URL to the .env file.
Problem 2 – Prisma Migration Drift After Schema Changes

After modifying schema.prisma mid-development to add the AuditLog table, running prisma migrate dev failed with "drift detected – the database schema is not in sync with the migration history." This happened because the Supabase database had been manually altered during testing.

Fix: Ran prisma migrate reset to wipe and re-apply all migrations cleanly, then re-ran prisma db seed to restore all seed data.
Problem 3 – Admin Session Lost on Every Page Refresh

After implementing JWT login, the admin was logged out every time the browser was refreshed. The token was being stored in a React state variable, which resets on reload.

Fix: Moved token storage to localStorage and updated AuthContext.jsx to read the token from localStorage on initial mount using useState(() => localStorage.getItem('token')).
Problem 4 – useEffect Causing an Infinite Re-Render Loop

The order queue page was constantly re-fetching and re-rendering in an endless loop, freezing the browser tab. An object (the headers config for Axios) was being created inside the component and passed into the useEffect dependency array, causing it to be re-evaluated as a new object on every render.

Fix: Moved the Axios config object outside the component scope and used only primitive values (the token string) in the dependency array.
Problem 5 – Environment Variables Undefined in Vite Frontend

After setting API_URL in a .env file, accessing process.env.API_URL in the React code returned undefined. Vite does not use process.env the same way Create React App does.

Fix: Renamed the variable to VITE_API_URL (Vite requires the VITE_ prefix) and accessed it using import.meta.env.VITE_API_URL in the frontend code.
Problem 6 – React Router v6 Syntax Breaking Tutorial Code

Following online tutorials for React Router resulted in errors because most tutorials were written for v5, which uses <Switch> and component={Page}. React Router v6 replaced these with <Routes> and element={<Page />}.

Fix: Rewrote all route definitions following the official React Router v6 documentation and updated the ProtectedRoute wrapper component to use the v6 Navigate component instead of Redirect.
